# Public Surface Guardrails

This document generalizes lessons from the TalkOrigins modernization work into
SciSiteForge framework policy. Site repos should still own their concrete route
maps, source inventories, and deployment commands, but these guardrails are
reusable across SciSiteForge-backed public sites.

The goal is to keep mature static sites stable, searchable, portable, and
reviewable while avoiding accidental broad rebuilds or search-facing drift.

## Update Scope Policy

Updates should be scoped to the smallest content class that actually changed.
This prevents unrelated regenerated pages from losing reviewed enrichment such
as expanded bibliographies, route-class metadata, search documents, or page
repair work.

Recommended scopes:

- `route`: one public page, plus explicitly named associated pages
- `route-class`: a content class such as bibliography, claim records, feedback,
  downloads, Notebook pages, or translation status
- `metadata`: title, description, canonical URL, JSON-LD, sitemap, and feed
  changes that do not alter article body text
- `search`: public search corpus files and search manifests
- `translation-status`: queue/status pages, manifests, and progress data
- `bibliography`: bibliography landing pages, topic pages, citation exports,
  BibTeX files, and associated feeds
- `portable`: offline/static distribution pages, manifests, checksums, and zip
  artifacts
- `global`: shell, theme, navigation, or framework changes that intentionally
  affect the whole public tree

A publish wrapper should refuse global updates unless the operator explicitly
selects a global scope. It should also refuse to publish sensitive route
classes through an overly broad generic route mode when that would bypass
class-specific safeguards.

## Canonical and Structured Metadata

Every public HTML page generated by SciSiteForge-compatible tooling should have
one absolute HTTPS canonical URL unless the page is intentionally excluded from
public indexing.

Generated metadata should be conservative:

- one `<link rel="canonical">` per public page
- one clear `<title>` and one non-empty meta description
- JSON-LD only for facts represented by the page
- stable `url` fields that match the canonical URL
- explicit content-kind metadata where the site uses route-class behavior
- a sitemap generated from canonical public routes

JSON-LD should support discovery and disambiguation. It should not be used to
make claims the page does not support.

## Search Corpus Contracts

Search should be treated as a public data product, not just a scrape of rendered
HTML.

Recommended contract:

- separate corpora by content class
- write a corpus manifest listing each JSONL file, document count, and kind
- reject malformed JSONL and duplicate route/key records
- reject blank text records
- warn on very short records and boilerplate-only records
- keep search-only rebuilds scoped to search files and associated metadata

Search corpus audits should run before publishing search updates. If direct
public URLs for search files are expected, deployment checks should verify that
the web server exposes those files, not just that rsync copied them.

SciSiteForge builds write `build/public_surface_guardrails.json` as a reusable
first-pass audit surface. The report currently validates HTML metadata,
JSON-LD parseability and URL consistency, active/process-surface separation,
and basic search corpus manifest/JSONL shape. Site repos should layer their own
route inventory, thresholds, and deployment checks on top of that framework
report.

## Bibliography Preservation

Expanded bibliography output is high-value generated content and should be
protected as its own route class.

Recommended safeguards:

- bibliography pages should have their own route-class publish scope
- generic route publishing should refuse bibliography paths unless explicitly
  allowed by bibliography policy
- build manifests should record restored pages, topic pages, citation counts,
  source links, DOI links, abstracts, BibTeX blocks, and BibTeX files
- bibliography audits should verify the expanded pages, topic exports, and
  BibTeX files before publish

This prevents later unrelated rebuilds from silently replacing enriched
bibliography pages with older or partial output.

## Education and Reader Paths

Reader-path features should be route-class aware and conservative. Related
education, Notebook, claim, bibliography, or dossier links can help search
engines and readers understand the site structure, but they should not rewrite
or reframe preserved legacy article bodies.

Recommended policy:

- add reader-path cards outside preserved attributed body text
- validate card titles, targets, descriptions, and search links
- treat pages with relevant terms but missing reader paths as warnings or
  backlog candidates, not automatic rewrites
- keep route-class updates scoped to the affected pages and their landing or
  metadata pages

## Portable and Offline Builds

Portable builds should be explicit public artifacts with their own metadata,
checksums, and exclusion policy.

Recommended contract:

- public-files-only input tree
- clear included/excluded content statement
- versioned release filename plus stable latest filename
- checksum files for stable and timestamped artifacts
- generated timestamp and package size on the download page
- offline search behavior documented where applicable
- private workbench, model logs, raw source extraction, and unreviewed data
  excluded by policy

## Performance Guardrails

SciSiteForge sites should retain the static, low-JavaScript reference-library
posture unless a site explicitly opts into an app surface.

Recommended audit checks:

- HTML size per page, with route-class exceptions for intentional large pages
- script tag count
- external script count
- stylesheet count
- inline script and style block size
- missing stylesheet warnings
- pages with many images lacking width or height attributes

Regressions that introduce unexpected JavaScript, many external scripts, or
oversized inline blocks should be errors. Legacy page size and image-dimension
issues can start as warning-level backlog items.

## Archive and Active Surface Separation

Archive pages, active status pages, review workbench pages, and operational
dashboards should be visibly distinct in metadata, copy, search corpora, and
route class.

Recommended content kinds:

- canonical archive article
- generated landing/navigation page
- bibliography topic
- claim/evidence record
- Notebook or reader-path page
- translation status
- public feedback archive
- public download artifact
- private or editorial workbench page

Search and JSON-LD should not blur private/editorial machinery into canonical
article content.

## GitHub Pages Publish Hygiene

GitHub Pages repositories are public distribution surfaces, not build
workspaces. Deployment tooling must copy only reviewed public artifacts into a
Pages checkout.

Default exclusions for GitHub Pages publishes:

- secrets, credentials, keys, tokens, cookies, session files, and environment
  files
- local absolute paths, usernames, hostnames, private IPs, and command logs
- generated build reports, route inventories, redirect plans, quality reports,
  search-corpus staging files, and deployment verification outputs unless they
  are intentionally public release artifacts
- private or editorial workbench pages, review dashboards, notebooks used only
  for operations, and migration planning surfaces
- model logs, translation run manifests, queue leases, node state, and
  machine-local progress state
- source snapshots that include dynamic app internals, password-reset routes,
  module paths, or other legacy attack-surface metadata

Publish wrappers should implement these exclusions directly rather than relying
on operator memory. Use allowlists or narrowly scoped rsync/git add rules when
possible, and treat broad `--delete` syncs into a Pages checkout as unsafe
unless exclusions are explicit and tested.

Before pushing a Pages update, deployment tooling should scan the staged tree
for common secret markers and local metadata, and should verify that disallowed
route classes are absent from the commit. If a bad Pages commit is pushed,
deleting files in a follow-up commit is not enough: rewrite the public branch
history with a clean replacement commit and force-push with lease. Credential
rotation is required if actual secrets were exposed; if only operational
metadata was exposed, record the incident and scrub history according to the
site's risk tolerance.

## Persistent Memory and Decision Records

When a site uses GroundRecall or another persistent memory system, major
publish-policy changes should be recorded as grounded decision notes.

Record at least:

- policy name and date
- affected site repo and framework repo
- implemented scripts or docs
- validation commands and results
- deployment result or explicit "no deployment needed"
- known warnings and backlog items

Persistent notes are not a substitute for versioned docs, but they keep
operational context available across long-running modernization work.

## Site-Local Versus Framework Ownership

Move an idea into SciSiteForge when it is a reusable contract or public-surface
pattern. Keep it in a site repo when it depends on one site's corpus,
deployment target, route inventory, or editorial backlog.

Framework-owned examples:

- update scope taxonomy
- metadata and canonical requirements
- search corpus contract
- performance audit categories
- portable artifact contract
- public/private route-class separation

Site-owned examples:

- actual canonical route map
- class-specific thresholds
- deployment host and rsync targets
- bibliography source provenance
- content-specific reader-path mappings
- warning allowlists and release-specific audit results
